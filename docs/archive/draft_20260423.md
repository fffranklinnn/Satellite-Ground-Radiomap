# 2. 分阶段修改路线图

仓库其实已经不是“纯旧系统”了，[`FrameContext`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/frame_context.py)、[`CoverageSpec`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/coverage_spec.py)、[`EntryWaveState/TerrainState/UrbanRefinementState`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/layer_states.py)、[`MultiScaleMap`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/multiscale_map.py)、[`ProductManifest`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/products/manifest.py) 都已经出现了。问题不在“有没有抽象”，而在“这些抽象是否已经真正接管物理语义”。

## 阶段1

- 目标  
统一时间、几何、坐标和 coverage 语义，只改接口和契约，不先改物理公式。

- 为什么先做这一阶段  
因为当前最大问题不是公式精度，而是“不同层输出的 256×256 数组是否真代表同一地理区域”。如果这一点不收敛，后面所有 benchmark、数据集、论文图都不稳。

- 修改内容  
重做 `CoverageSpec`，让它显式区分 `l1_grid / l2_grid / l3_grid / product_grid`；弱化 `FrameContext.grid` 的单一网格语义；统一所有入口脚本到 `parse_iso_utc()`；禁止主线脚本继续使用裸 `0.256` 和脚本内局部时间解析。

- 新增抽象  
建议把当前 `CoverageSpec` 升级为：
`CoverageSpec(l1_grid, l2_grid, l3_grid, product_grid, alignment_rule, projection_rule, blend_rule, version)`  
同时给 `FrameContext` 加只读属性：
`frame.l1_grid`, `frame.l2_grid`, `frame.l3_grid`, `frame.product_grid`。

- 对现有代码的最小侵入式改法  
保留现有 `BaseLayer.compute(origin_lat, origin_lon, ...)` 作为 legacy path；新增 canonical 访问方式只从 `frame.coverage` 取网格。先不删 `frame.grid`，但把它标注为 deprecated alias，不允许在新主线里再使用。

- 验收标准  
同一帧下，L1/L2/L3/product 的 bbox、中心点、分辨率都能被显式打印和序列化；主线代码中不再存在“把不同尺度数组仅因 shape 相同就视作同网格”的路径；所有脚本入口时间统一走 UTC helper。

- 风险  
会暴露很多现有测试其实只验证 shape，没有验证 footprint；短期内会让一批“以前能跑”的脚本先报契约错误。

## 阶段2

- 目标  
让 L1/L2/L3 都绑定同一个 `FrameContext`，解决层间不同步问题。

- 为什么先做这一阶段  
阶段1解决“地图语义”，阶段2解决“链路语义”。必须让“某时刻、某颗星、某区域”在进入各层前就已经确定，而不是由 L1 选完星之后再让 L2/L3 跟着猜。

- 修改内容  
把“选星”从 `L1.compute_components()` 内部抽出来，变成 frame 生成过程的一部分；主线改成“先选星，再 build frame，再逐层传播”。`L2/L3` 在 canonical path 下不再允许依赖 `LayerContext.extras` 注入几何。

- 新增抽象  
可引入轻量 `FramePlan` 或 `SelectedLink`，字段只需包含：
`timestamp_utc, norad_id, sat_lat_deg, sat_lon_deg, sat_alt_m, center_azimuth_deg, center_elevation_deg, center_slant_range_m`。  
如果你不想再加新对象，也可以直接把这组字段强制前置进 `FrameContext`。

- 对现有代码的最小侵入式改法  
最小改法不是重写 L1，而是把 `L1._select_best_satellite()` 封成 `select_satellite_for_frame()`；`main.py`、`BenchmarkRunner`、多星脚本都先调它，再把结果传给 `FrameBuilder.build(timestamp, sat_info=...)`。`propagate_entry()` 不再负责“决定是哪颗星”，只负责“在已确定链路上做 L1 传播”。

- 验收标准  
同一 frame 的 `norad_id / azimuth / elevation / slant_range` 在 `FrameContext`、`EntryWaveState`、`TerrainState`、`UrbanRefinementState` 和 manifest 中一致；主线不再出现“frame 无 sat_info，但层内再选星”的路径。

- 风险  
多星脚本需要“一星一 frame”或“一 frame 多 link”的明确策略；这一步会让一部分“临时通过 `target_norad_ids` 注入”的脚本需要重写入口。

## 阶段3

- 目标  
重构多尺度表示，明确 30m coarse map 与 1m urban refinement 的关系。

- 为什么先做这一阶段  
这是项目从“三层平行脚本”升级到“多尺度模拟器”的核心。L3 不应再作为与 L2 平行的全图层，而应是局部 urban residual / local refinement。

- 修改内容  
让 `EntryWaveState`、`TerrainState`、`UrbanRefinementState` 都保留各自 `native_grid`；新增统一投影步骤 `project_to_product_grid()`；`MultiScaleMap` 不再直接接收“已经假定同网格”的数组，而是负责“native state -> aligned product view -> blend”。

- 新增抽象  
建议新增：
`ProjectedLayerView(grid, values_db, source_state, projection_rule)`  
或更简化一点，在 `MultiScaleMap` 内部维护：
`projected_l1_db, projected_l2_db, projected_l3_db, support_mask, blend_policy`。

- 对现有代码的最小侵入式改法  
先不改 L2/L3 的内部物理计算，只改它们的 state 语义：
`TerrainState.grid` 改成 `native_grid=frame.l2_grid`  
`UrbanRefinementState.grid` 改成 `native_grid=frame.l3_grid`  
然后新增一个 projector/resampler，把它们投影到 `product_grid` 后再组合。  
也就是说，先改“投影与拼接层”，后改“物理求解层”。

- 验收标准  
产品图的组合公式变成“投影后组合”而不是“shape 相同直接相加”；非 urban 区域中，最终产品等于 coarse backbone；城市 tile 中只在 support 区域叠加 urban residual；L2/L3 的 native coverage 在 manifest 中可追溯。

- 风险  
会增加一层 resample/crop 逻辑，性能和内存会上升；如果不先定义好 projection protocol，很容易把 phase 3 再做成“新的隐藏插值”。

建议的 canonical pipeline 伪代码如下：

```python
sat = select_satellite_for_frame(timestamp=ts, region=center, target_ids=target_ids)
frame = frame_builder.build(timestamp=ts, sat_info=sat, coverage=coverage, strict=True)

entry = l1.propagate_entry(frame, grid=frame.l1_grid)
terrain = l2.propagate_terrain(frame, entry=entry, grid=frame.l2_grid)
urban = l3.refine_urban(frame, entry=entry, grid=frame.l3_grid)

product = compose_to_product(
    frame=frame,
    entry=entry,
    terrain=terrain,
    urban=urban,
    target_grid=frame.product_grid,
    blend_rule="coarse_plus_masked_residual",
)
```

## 阶段4

- 目标  
加入 manifest / provenance / strict UTC / strict snapshot / benchmark mode。

- 为什么先做这一阶段  
前 3 阶段解决“是不是同一个物理场景”，这一阶段解决“这个场景是否可严格复现”。

- 修改内容  
把 `ProductManifest` 从“输出记录器”升级为“实验契约”；在 benchmark mode 下，缺失数据、naive 时间、fallback 路径都直接失败；对输入 snapshot 做稳定哈希，而不是仅靠手填 `snapshot_id`。

- 新增抽象  
建议给 `ProductManifest` 增加：
`benchmark_id, schema_version, input_snapshot_hash, coverage_signature, frame_contract_hash, fallback_policy, software_version/git_commit`。  
建议给 pipeline 增加：
`BenchmarkMode(strict_utc=True, strict_snapshot=True, allow_fallback=False, tolerance=...)`。

- 对现有代码的最小侵入式改法  
保留现有 manifest 结构，但先加字段，不先改导出格式；`main.py` 继续普通模式，`BenchmarkRunner` 先切成 strict canonical path；所有脚本入口统一调用 `parse_iso_utc(strict=...)`。

- 验收标准  
同一 config + 同一 data snapshot + 同一 frame list，输出 manifest 完全一致；缺失 IONEX/ERA5/DEM/L3 tile 时 benchmark mode 直接失败；manifest 可明确区分“正常数据”和“fallback 运行”。

- 风险  
一旦 strict mode 真正生效，当前仓库里不少“方便运行”的路径会停止工作；但这是正确代价。

## 阶段5

- 目标  
再考虑电离层、极化、多种地图产品、数据集导出和后续 AI 接口。

- 为什么先做这一阶段  
因为这时你的系统已经先具备了科学一致性和可复现性，后续物理增强和数据集扩展才不会把旧歧义带进来。

- 修改内容  
把 L1 分解出的物理子项正式纳入产品层；支持 path loss 之外的 `visibility`, `entry angle`, `terrain blockage`, `urban residual`, `component maps`, `timeseries cube` 等产品；为 AI 接口导出 typed sample，而不是 PNG 结果图。

- 新增抽象  
建议显式引入 `MapProduct`：
`product_id, product_type, grid, values, units, source_states, manifest_ref`。  
AI 侧只消费 `MapProduct + ProductManifest`，不直连层内部实现。

- 对现有代码的最小侵入式改法  
复用现有 [`projectors.py`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/products/projectors.py) 和 `EntryWaveState` 的已有分量字段，不改现有物理公式，只扩展导出路径。

- 验收标准  
一次 frame-run 可以稳定导出多种产品；数据集样本可追溯到 frame、coverage、satellite geometry、snapshot 和 fallbacks；后续 AI 训练不需要回头解析脚本输出图。

- 风险  
很容易范围失控，尤其是“先做 AI 接口”会反过来绑架底层设计。这里必须坚持“先物理契约，后数据产品”。

# 3. 核心数据结构建议

- `FrameContext`  
建议保留，但升级。它应该是“单帧传播契约”，不是“单个 grid 容器”。  
建议字段：`frame_id`, `timestamp_utc`, `coverage`, `region_id`, `scene_center_lat`, `scene_center_lon`, `norad_id`, `sat_lat_deg`, `sat_lon_deg`, `sat_alt_m`, `center_azimuth_deg`, `center_elevation_deg`, `center_slant_range_m`, `snapshot_id`, `strict_flags`, `tags`。  
建议：`grid` 不再作为唯一空间语义入口，改为兼容属性，主线统一走 `coverage`。

- `GridSpec`  
应该保留，并继续保持 immutable。  
建议字段：`grid_id`, `role`(`l1/l2/l3/product`), `crs`, `anchor_type`, `anchor_lat`, `anchor_lon`, `width_m`, `height_m`, `nx`, `ny`, `dx_m`, `dy_m`, `pixel_registration`, `row_order`, `col_order`, `bbox`, `source_resolution_m`, `semantic_note`。  
建议：`role` 很重要，因为同样是 256×256，语义完全可能不同。

- `CoverageSpec`  
必须升级，当前版本过于简化。  
建议字段：`l1_grid`, `l2_grid`, `l3_grid`, `product_grid`, `alignment_rule`, `projection_rule`, `crop_rule`, `blend_rule`, `urban_support_rule`, `version`。  
不建议继续只有一个 `coarse_grid`，因为它在这个项目里会把 L1 和 L2 错误地折叠成同一尺度。

- `EntryWaveState`  
应该保留，它已经是很好的基础。  
建议字段：`frame_id`, `native_grid`, `total_loss_db`, `fspl_db`, `atm_db`, `iono_db`, `pol_db`, `antenna_gain_db`, `elevation_deg`, `azimuth_deg`, `slant_range_m`, `visibility_mask`, `norad_id`, `sat_lat_deg`, `sat_lon_deg`, `sat_alt_m`, `metadata`。  
建议：把 `grid` 明确改成 `native_grid`，避免被误解成 product grid。

- `TerrainState`  
必须保留，但要修正语义。  
建议字段：`frame_id`, `native_grid`, `terrain_loss_db`, `true_occlusion_mask`, `diffraction_loss_db`, `dem_elevation_m`, `horizon_metric`, `padding_m`, `dem_source_id`, `metadata`。  
建议：不要再用 `loss_db >= max_loss` 反推遮挡；真实遮挡 mask 要从 L2 计算链直接返回。

- `UrbanRefinementState`  
应该保留，并明确它是 residual。  
建议字段：`frame_id`, `native_grid`, `urban_residual_db`, `support_mask`, `nlos_mask`, `building_height_m`, `occupancy_mask`, `incident_azimuth_deg`, `incident_elevation_deg`, `tile_ids`, `coverage_fraction`, `metadata`。  
建议：`support_mask` 表达“urban refinement 生效的空间支持域”，不要只是“tile 读成功就全 1”。

- `MultiScaleMap`  
应该保留，但职责要变。它不应再是“同 shape 数组求和器”，而应是“多尺度组合记录”。  
建议字段：`frame_id`, `product_grid`, `projected_l1_db`, `projected_l2_db`, `projected_l3_db`, `composite_db`, `support_masks`, `projection_metadata`, `blend_policy`, `source_state_refs`。  
建议：组合发生在 `product_grid` 上，不发生在 native grids 上。

- `ProductManifest`  
应该保留，并升格为科研 provenance 核心对象。  
建议字段：`product_id`, `frame_id`, `timestamp_utc`, `product_type`, `config_hash`, `input_snapshot_hash`, `input_file_hashes`, `output_file_hashes`, `coverage_signature`, `frame_contract_hash`, `fallbacks_used`, `benchmark_id`, `schema_version`, `software_version/git_commit`, `notes`。  
建议：benchmark mode 下，缺失输入不应再被静默省略。

# 6. 不要做的事情

- 不要一开始就重写全部 L1/L2/L3 物理公式，当前优先级是统一语义，不是换模型。

- 不要继续使用语义不清的 `origin`；至少要显式区分 `center` 和 `sw_corner`。
- 不要在 benchmark / strict mode 下保留静默 fallback。
- 不要把 PNG 输出当作数据集或 benchmark 的事实标准，应该导出 typed product arrays + manifest。
- 不要在 phase 3 之前就把 AI 接口设计绑定到底层传播状态上，先把物理契约收敛。
