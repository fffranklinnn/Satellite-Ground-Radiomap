# Satellite-Ground-Radiomap Semantic Unification & Multi-Scale Refactoring Plan

## Goal Description

Refactor the Satellite-Ground-Radiomap codebase to fix semantic inconsistencies across the three-layer propagation pipeline (L1 macro/atmosphere, L2 terrain/topography, L3 urban refinement). The core problems are: (1) CoverageSpec collapses L1/L2/L3 into a single grid, causing "same shape = same grid" composition errors; (2) satellite selection happens inside L1 instead of at frame level, breaking inter-layer consistency; (3) multi-scale composition adds arrays by shape without projection; (4) provenance/reproducibility is not strict enough for benchmark-grade science. The refactoring proceeds in 5 phases: coverage/coordinate unification, frame-level satellite binding, projection-based composition, strict provenance, and multi-product export.

## Acceptance Criteria

Following TDD philosophy, each criterion includes positive and negative tests for deterministic verification.

- AC-1: Multi-scale grid semantic correctness
  - Positive Tests (expected to PASS):
    - A built frame exposes validated `coverage.l1_grid`, `coverage.l2_grid`, `coverage.l3_grid`, `coverage.product_grid` with explicit anchor, extent, resolution, and role metadata
    - Each grid's bbox, center, and resolution can be serialized to JSON and round-tripped without loss
    - All grids share the same center point (concentric constraint)
    - `l2_grid` bbox is contained within `l1_grid` bbox; `l3_grid` within `l2_grid`
  - Negative Tests (expected to FAIL):
    - Constructing a CoverageSpec where `l2_grid` center differs from `l1_grid` center raises a validation error
    - Constructing a CoverageSpec where `l3_grid` extent exceeds `l2_grid` extent raises a validation error
    - Accessing deprecated `frame.grid` on the canonical strict path emits a deprecation warning
  - AC-1.1: UTC contract unification
    - Positive: All script entry points parse timestamps via `parse_iso_utc()` from `time_utils`
    - Negative: A script using `datetime.fromisoformat()` directly (without `parse_iso_utc`) fails linting/contract checks

- AC-2: Frame-level satellite binding
  - Positive Tests (expected to PASS):
    - `SatelliteSelector.select()` returns a deterministic result: highest elevation wins, tie-break by lowest NORAD ID
    - Same frame's `norad_id`, `azimuth_deg`, `elevation_deg`, `slant_range_m` are identical in `FrameContext`, `EntryWaveState`, `TerrainState`, `UrbanRefinementState`, and `ProductManifest`
    - `FrameBuilder.build(timestamp, sat_info=...)` on canonical path produces a frame with satellite geometry pre-bound
  - Negative Tests (expected to FAIL):
    - Calling `L1MacroLayer.propagate_entry()` on canonical path without pre-selected `sat_info` in frame raises an error
    - `SatelliteSelector.select()` with no visible satellites in strict mode raises an error (not silent fallback)
    - `target_norad_ids` filtering with an invalid NORAD ID raises a validation error

- AC-3: Projection-based multi-scale composition
  - Positive Tests (expected to PASS):
    - `MultiScaleMap.compose()` accepts projected layer views on `product_grid`, not raw native-grid arrays
    - Non-urban areas in the final product equal the coarse (L1+L2) backbone exactly
    - Urban tiles add residual only within the `support_mask` region
    - `TerrainState.occlusion_mask` is derived from geometric occlusion computation, not from `loss_db >= threshold`
    - `UrbanRefinementState.support_mask` reflects actual tile data availability, not unconditional all-ones
  - Negative Tests (expected to FAIL):
    - Passing raw native-grid arrays (without projection) to `MultiScaleMap.compose()` raises a type/contract error
    - Composing arrays with matching shape but different grid metadata raises a grid-mismatch error
  - AC-3.1: Projection contract per field type
    - Positive: Loss fields (dB) use bilinear interpolation; boolean masks use nearest-neighbor; angle fields handle azimuth wrap
    - Negative: Projecting a boolean mask with bilinear interpolation raises a contract error

- AC-4: Strict provenance and benchmark mode
  - Positive Tests (expected to PASS):
    - Same config + same data snapshot + same frame list produces identical `ProductManifest` output
    - `ProductManifest.provenance` block includes `benchmark_id`, `schema_version`, `input_snapshot_hash`, `coverage_signature`, `frame_contract_hash`, `software_version`
    - Benchmark mode with missing configured IONEX/ERA5/DEM/L3 tile data raises a hard failure
    - Manifest distinguishes "normal data" runs from "fallback" runs via `fallbacks_used` field
  - Negative Tests (expected to FAIL):
    - In benchmark mode, a naive (non-UTC) timestamp raises `StrictModeError`
    - In benchmark mode, a configured but missing input file raises an error (not silently omitted)
    - In benchmark mode, an empty `data_snapshot_id` raises a validation error
  - AC-4.1: Provenance scoping
    - Positive: In normal mode, unconfigured optional inputs are silently skipped; configured but missing inputs emit warnings
    - Negative: In normal mode, unconfigured optional inputs do NOT raise errors

- AC-5: Multi-product export (future enhancement)
  - Positive Tests (expected to PASS):
    - A single frame-run can export multiple product types (path loss, visibility, terrain blockage, urban residual)
    - Dataset samples are traceable to frame, coverage, satellite geometry, snapshot, and fallbacks
    - Existing `projectors.py` product registry is reused for incremental product growth
  - Negative Tests (expected to FAIL):
    - Exporting a product without a valid `ProductManifest` reference raises an error
    - AI interface consuming layer internals directly (bypassing product abstraction) is not supported in canonical path

## Path Boundaries

Path boundaries define the acceptable range of implementation quality and choices.

### Upper Bound (Maximum Acceptable Scope)
The implementation completes Phases 1-4 with full test coverage: CoverageSpec upgraded to 4-grid with formal geometry contract; SatelliteSelector service with deterministic selection; projection-based composition with per-field-type contracts; strict provenance with nested versioned manifest block; all mainline scripts migrated to canonical frame/state APIs; golden-scene regression tests for grid geometry, selection determinism, projected composition, and provenance reproducibility; legacy scripts moved to `scripts/legacy/` with adapter wrappers. Phase 5 (multi-product export) is designed but not fully implemented.

### Lower Bound (Minimum Acceptable Scope)
The implementation completes Phases 1-3 core semantics: CoverageSpec exposes 4 grids under `frame.coverage`; satellite selection extracted from L1 into shared service; layer states carry `native_grid`; `MultiScaleMap.compose()` requires projected inputs; `support_mask` and `occlusion_mask` fixed. `frame.grid` deprecated with warnings. UTC unified in `main.py` and `BenchmarkRunner`. Phase 4 provenance fields added but strict enforcement limited to benchmark path. Legacy scripts still functional via compatibility aliases.

### Allowed Choices
- Can use: existing `BlendPolicy` enum for composition rules; existing `GridSpec` as immutable grid descriptor; bilinear/nearest-neighbor interpolation for projection; frozen dataclasses for state objects; `@property` for deprecated aliases
- Can use: enum or string constants for grid roles (`l1_macro`, `l2_terrain`, `l3_urban`, `product`)
- Cannot use: time estimates, line-number references, or plan-specific terminology in implementation code
- Cannot use: rewriting L1/L2/L3 physics formulas (semantic repair only, not model changes)
- Cannot use: PNG as benchmark/dataset output format (typed arrays + manifest required)
- Cannot use: `frame.grid` as sole spatial semantic entry in new canonical code

### Conceptual Approach

The canonical pipeline after refactoring:

```python
# Phase 2: Satellite selection at frame level
selector = SatelliteSelector(tle_path=config["tle"], strict=strict)
sat = selector.select(timestamp=ts, center=(lat, lon), target_ids=target_ids)

# Phase 1: Frame with 4-grid coverage
frame = frame_builder.build(timestamp=ts, sat_info=sat, coverage=coverage, strict=True)
# frame.coverage.l1_grid, frame.coverage.l2_grid, frame.coverage.l3_grid, frame.coverage.product_grid

# Layer propagation on native grids
entry = l1.propagate_entry(frame, grid=frame.coverage.l1_grid)
terrain = l2.propagate_terrain(frame, entry=entry, grid=frame.coverage.l2_grid)
urban = l3.refine_urban(frame, entry=entry, grid=frame.coverage.l3_grid)

# Phase 3: Project then compose
product = compose_to_product(
    frame=frame,
    entry=entry,
    terrain=terrain,
    urban=urban,
    target_grid=frame.coverage.product_grid,
    blend_rule="coarse_plus_masked_residual",
)
```

### Formal Geometry Contract

- Grid roles: `l1_macro` (256 km, ~1000 m/px), `l2_terrain` (25.6 km, ~100 m/px), `l3_urban` (256 m, ~1 m/px), `product` (explicit composition target)
- Anchor: center-anchored for all roles; L2 SW-corner convention eliminated from canonical path
- CRS: WGS84 geographic (lat/lon degrees)
- Concentricity: all grids share the same center point; extent and resolution vary by role
- Nesting: `l2` bbox contained within `l1` bbox; `l3` within `l2`; `product_grid` is the explicit composition/export target with a declared crop rule (may be encompassing or centered crop, versioned in the contract)
- Each layer owns its native grid; composition happens on `product_grid` only

### Projection Contract Per Field Type

| Field Type | Interpolation | Notes |
|------------|--------------|-------|
| Loss fields (dB) | Bilinear | Standard resampling for continuous dB values |
| Boolean masks (occlusion, nlos) | Nearest-neighbor | Preserves binary semantics |
| Support masks | Nearest-neighbor + conservative AND at boundaries | Prevents false positive support at edges |
| Angle fields (azimuth) | Bilinear with wrap handling | Handle 0°/360° discontinuity |
| Angle fields (elevation) | Bilinear | No wrap needed |
| Visibility masks | Nearest-neighbor | Binary field |

## Claude-Codex Deliberation

### Agreements
- Phase 1 addresses a real semantic break: CoverageSpec exists but composition ignores grid roles and BlendPolicy
- Phase 2 is directionally correct: satellite selection must be fixed before propagation, not re-selected inside L1
- Phase 3 is necessary: MultiScaleMap.compose() is shape-based addition; support_mask and occlusion_mask issues are real
- Phase 4 is justified: current provenance is insufficient for reproducible benchmarking
- Legacy script debt is real: generate_xian_city_radiomap.py and generate_feature_showcase.py carry workarounds conflicting with newer pipeline
- Adding golden fixtures before semantic refactors is the right sequencing
- Migration/compatibility plan with deprecated aliases reduces breakage
- Phase 5 scoped as future enhancement is correct; geometry/projection contracts must settle first

### Resolved Disagreements
- **sat_info mandatory timing**: Claude initially proposed making sat_info mandatory immediately. Codex argued this is too aggressive. Resolution: mandatory only on canonical strict path; legacy path retains optional with deprecation warning. Rationale: reduces migration risk while establishing the correct contract.
- **Selector abstraction level**: Claude initially proposed a bare standalone function. Codex argued a selector service/module is safer due to TLE state, filters, and multisat complexity. Resolution: shared SatelliteSelector service in src/planning/. Rationale: encapsulates selection concerns cleanly.
- **Provenance timing**: Claude initially placed all provenance work in Phase 4. Codex argued minimal regression fixtures should exist before Phase 2/3 refactors. Resolution: golden-scene test fixtures added to each phase before semantic changes. Rationale: provides stable baseline for detecting regressions.
- **Phase 5 scope**: Claude included full MapProduct + AI interface. Codex argued this is over-scoped as convergence blocker. Resolution: Phase 5 marked as future enhancement, reusing existing projectors.py. Rationale: core semantic repair is the priority.
- **product_grid containment**: Claude initially required product_grid to "encompass all" native grids. Codex argued this is too rigid given existing cropped-target workflows. Resolution: product_grid is explicit composition target with declared crop rule (encompassing or centered crop), versioned in contract. Rationale: preserves urban-detail usefulness.

# 2. 分阶段修改路线图

仓库其实已经不是“纯旧系统”了，[`FrameContext`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/frame_context.py)、[`CoverageSpec`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/coverage_spec.py)、[`EntryWaveState/TerrainState/UrbanRefinementState`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/layer_states.py)、[`MultiScaleMap`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/context/multiscale_map.py)、[`ProductManifest`](/home/Users_Work_Space/zsfang/Satellite-Ground-Radiomap/src/products/manifest.py) 都已经出现了。问题不在“有没有抽象”，而在“这些抽象是否已经真正接管物理语义”。

## 阶段1

- 目标  
统一时间、几何、坐标和 coverage 语义，只改接口和契约，不先改物理公式。

- 为什么先做这一阶段  
因为当前最大问题不是公式精度，而是“不同层输出的 256×256 数组是否真代表同一地理区域”。如果这一点不收敛，后面所有 benchmark、数据集、论文图都不稳。

- 修改内容  
重做 `CoverageSpec`，让它显式区分 `l1_grid / l2_grid / l3_grid / product_grid`；弱化 `FrameContext.grid` 的单一网格语义；统一所有入口脚本到 `parse_iso_utc()`；禁止主线脚本继续使用裸 `0.256` 和脚本内局部时间解析。

- 新增抽象  
建议把当前 `CoverageSpec` 升级为：
`CoverageSpec(l1_grid, l2_grid, l3_grid, product_grid, alignment_rule, projection_rule, blend_rule, version)`  
同时给 `FrameContext` 加只读属性：
`frame.l1_grid`, `frame.l2_grid`, `frame.l3_grid`, `frame.product_grid`。

- 对现有代码的最小侵入式改法  
保留现有 `BaseLayer.compute(origin_lat, origin_lon, ...)` 作为 legacy path；新增 canonical 访问方式只从 `frame.coverage` 取网格。先不删 `frame.grid`，但把它标注为 deprecated alias，不允许在新主线里再使用。

- 验收标准  
同一帧下，L1/L2/L3/product 的 bbox、中心点、分辨率都能被显式打印和序列化；主线代码中不再存在“把不同尺度数组仅因 shape 相同就视作同网格”的路径；所有脚本入口时间统一走 UTC helper。

- 风险  
会暴露很多现有测试其实只验证 shape，没有验证 footprint；短期内会让一批“以前能跑”的脚本先报契约错误。

## 阶段2

- 目标  
让 L1/L2/L3 都绑定同一个 `FrameContext`，解决层间不同步问题。

- 为什么先做这一阶段  
阶段1解决“地图语义”，阶段2解决“链路语义”。必须让“某时刻、某颗星、某区域”在进入各层前就已经确定，而不是由 L1 选完星之后再让 L2/L3 跟着猜。

- 修改内容  
把“选星”从 `L1.compute_components()` 内部抽出来，变成 frame 生成过程的一部分；主线改成“先选星，再 build frame，再逐层传播”。`L2/L3` 在 canonical path 下不再允许依赖 `LayerContext.extras` 注入几何。

- 新增抽象  
可引入轻量 `FramePlan` 或 `SelectedLink`，字段只需包含：
`timestamp_utc, norad_id, sat_lat_deg, sat_lon_deg, sat_alt_m, center_azimuth_deg, center_elevation_deg, center_slant_range_m`。  
如果你不想再加新对象，也可以直接把这组字段强制前置进 `FrameContext`。

- 对现有代码的最小侵入式改法  
最小改法不是重写 L1，而是把 `L1._select_best_satellite()` 封成 `select_satellite_for_frame()`；`main.py`、`BenchmarkRunner`、多星脚本都先调它，再把结果传给 `FrameBuilder.build(timestamp, sat_info=...)`。`propagate_entry()` 不再负责“决定是哪颗星”，只负责“在已确定链路上做 L1 传播”。

- 验收标准  
同一 frame 的 `norad_id / azimuth / elevation / slant_range` 在 `FrameContext`、`EntryWaveState`、`TerrainState`、`UrbanRefinementState` 和 manifest 中一致；主线不再出现“frame 无 sat_info，但层内再选星”的路径。

- 风险  
多星脚本需要“一星一 frame”或“一 frame 多 link”的明确策略；这一步会让一部分“临时通过 `target_norad_ids` 注入”的脚本需要重写入口。

## 阶段3

- 目标  
重构多尺度表示，明确 30m coarse map 与 1m urban refinement 的关系。

- 为什么先做这一阶段  
这是项目从“三层平行脚本”升级到“多尺度模拟器”的核心。L3 不应再作为与 L2 平行的全图层，而应是局部 urban residual / local refinement。

- 修改内容  
让 `EntryWaveState`、`TerrainState`、`UrbanRefinementState` 都保留各自 `native_grid`；新增统一投影步骤 `project_to_product_grid()`；`MultiScaleMap` 不再直接接收“已经假定同网格”的数组，而是负责“native state -> aligned product view -> blend”。

- 新增抽象  
建议新增：
`ProjectedLayerView(grid, values_db, source_state, projection_rule)`  
或更简化一点，在 `MultiScaleMap` 内部维护：
`projected_l1_db, projected_l2_db, projected_l3_db, support_mask, blend_policy`。

- 对现有代码的最小侵入式改法  
先不改 L2/L3 的内部物理计算，只改它们的 state 语义：
`TerrainState.grid` 改成 `native_grid=frame.l2_grid`  
`UrbanRefinementState.grid` 改成 `native_grid=frame.l3_grid`  
然后新增一个 projector/resampler，把它们投影到 `product_grid` 后再组合。  
也就是说，先改“投影与拼接层”，后改“物理求解层”。

- 验收标准  
产品图的组合公式变成“投影后组合”而不是“shape 相同直接相加”；非 urban 区域中，最终产品等于 coarse backbone；城市 tile 中只在 support 区域叠加 urban residual；L2/L3 的 native coverage 在 manifest 中可追溯。

- 风险  
会增加一层 resample/crop 逻辑，性能和内存会上升；如果不先定义好 projection protocol，很容易把 phase 3 再做成“新的隐藏插值”。

建议的 canonical pipeline 伪代码如下：

```python
sat = select_satellite_for_frame(timestamp=ts, region=center, target_ids=target_ids)
frame = frame_builder.build(timestamp=ts, sat_info=sat, coverage=coverage, strict=True)

entry = l1.propagate_entry(frame, grid=frame.l1_grid)
terrain = l2.propagate_terrain(frame, entry=entry, grid=frame.l2_grid)
urban = l3.refine_urban(frame, entry=entry, grid=frame.l3_grid)

product = compose_to_product(
    frame=frame,
    entry=entry,
    terrain=terrain,
    urban=urban,
    target_grid=frame.product_grid,
    blend_rule="coarse_plus_masked_residual",
)
```

## 阶段4

- 目标  
加入 manifest / provenance / strict UTC / strict snapshot / benchmark mode。

- 为什么先做这一阶段  
前 3 阶段解决“是不是同一个物理场景”，这一阶段解决“这个场景是否可严格复现”。

- 修改内容  
把 `ProductManifest` 从“输出记录器”升级为“实验契约”；在 benchmark mode 下，缺失数据、naive 时间、fallback 路径都直接失败；对输入 snapshot 做稳定哈希，而不是仅靠手填 `snapshot_id`。

建议给 `ProductManifest` 增加：
`benchmark_id, schema_version, input_snapshot_hash, coverage_signature, frame_contract_hash, fallback_policy, software_version/git_commit`。  
建议给 pipeline 增加：
`BenchmarkMode(strict_utc=True, strict_snapshot=True, allow_fallback=False, tolerance=...)`。

- 对现有代码的最小侵入式改法  
保留现有 manifest 结构，但先加字段，不先改导出格式；`main.py` 继续普通模式，`BenchmarkRunner` 先切成 strict canonical path；所有脚本入口统一调用 `parse_iso_utc(strict=...)`。

- 验收标准  
同一 config + 同一 data snapshot + 同一 frame list，输出 manifest 完全一致；缺失 IONEX/ERA5/DEM/L3 tile 时 benchmark mode 直接失败；manifest 可明确区分“正常数据”和“fallback 运行”。

- 风险
